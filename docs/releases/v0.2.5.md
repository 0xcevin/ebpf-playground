## 新增功能

### 1. 启动时同时打印内核版本与系统版本

程序启动时的环境检测区块现在会输出两行：

```
=== eBPF Tracepoint 环境检测 ===
内核版本: 3.10.0-1160.el7.x86_64
系统版本: CentOS Linux 7 (Core)
...
```

系统版本通过读取 `/etc/os-release`（或 fallback 到 `/etc/redhat-release`、`/etc/centos-release`）获得，方便运维人员快速识别目标环境。

### 2. 按需加载 eBPF 对象

当使用 `-execve` 或 `-net` 参数时，程序现在会从 eBPF CollectionSpec 中**剔除**不需要的 program 和 map，再执行加载。

| 参数 | 保留对象 | 剔除对象 |
|------|---------|---------|
| `-execve` | `sys_enter_execve`、`sys_exit_execve`、`pb` | `sys_enter_connect`、`sys_enter_accept4`、`sys_exit_accept4`、`accept4_sockaddr` |
| `-net` | `sys_enter_connect`、`sys_enter_accept4`、`sys_exit_accept4`、`accept4_sockaddr`、`pb` | `sys_enter_execve`、`sys_exit_execve` |

**收益**：避免在老内核上因为加载未使用的 map（如 `BPF_MAP_TYPE_HASH`）而意外失败。

### 3. bpf syscall 可用性探测

在解除 memlock 之后，程序会尝试调用 `bpf(2)` 系统调用进行探测：

- 如果返回 `ENOSYS`（function not implemented），说明当前内核**没有 eBPF 支持**。
- 程序会立即退出并给出友好提示，而不是继续加载后报晦涩的 `map create: function not implemented`。

### 4. 最低内核版本拦截

程序现在会显式检查内核版本：

- **< 3.10**：直接退出，提示 `本工具不支持低于 Linux 3.10 的内核`。
- **3.10 (RHEL/CentOS/Oracle 7.6+ 回移植内核)**：允许进入兼容模式。
- **4.7+**：允许进入兼容模式（tracepoint 最低要求）。
- **5.8+**：进入现代模式（ringbuf + BTF）。

所有错误提示中的最低版本要求已统一为 **Linux 4.7+**。

## 文档更新

### 官方支持矩阵

README.md 与 CHANGELOG.md 已同步更新，明确列出以下常见发行版的支持状况：

| 发行版 | 支持版本 | 不支持版本 |
|--------|---------|-----------|
| CentOS / RHEL | 7.6+、8.x、9.x | 7.5 及以下 |
| Rocky Linux | 8.x、9.x | — |
| AlmaLinux | 8.x、9.x | — |
| Oracle Linux | 7.6+ (UEK)、8.x (UEK6) | 7.5 及以下 |
| Ubuntu | 18.04、20.04、22.04+ | — |
| Debian | 9、10、11、12 | — |

> **注意**：3.10 内核仅限 **RHEL 系发行版的回移植版本**（7.6+）才支持 eBPF tracepoint。上游 3.10 或其他发行版的 3.10 **不支持**。

## 0.2.x 历史变更（合并记录）

### v0.2.4（未单独发版）

v0.2.4 的变更已合并进 v0.2.5，包括环境检测增强、按需加载、发行版矩阵等。

### v0.2.3 — 修复 CentOS 7 上兼容模式 Map BTF 加载失败

**问题**：CentOS 7.5 (3.10.0-862) 上兼容模式报错 `load BTF: detect support for Map BTF (Var/Datasec): function not implemented`。  
**根因**：`cilium/ebpf` 发现 `MapSpec.Key` / `MapSpec.Value` 不为 nil，尝试检测内核 Map BTF 支持性，但 CentOS 7 的 3.10 内核不支持 Map BTF。  
**修复**：在 `legacy.go` 加载兼容对象前，清空所有 map 的 `Key` 和 `Value` BTF 字段，使 `cilium/ebpf` 以传统方式创建 map。

### v0.2.2 — 修复 CentOS 7 上 memlock 解除失败

**问题**：CentOS 7.5 (3.10.0-862) 上 `rlimit.RemoveMemlock()` 报错 `unexpected error detecting memory cgroup accounting: function not implemented`。  
**根因**：`cilium/ebpf` 的 `RemoveMemlock` 在内部检测 cgroup 内存会计时调用了老内核不支持的系统调用。  
**修复**：`RemoveMemlock` 失败时，降级到 `unix.Getrlimit` + `unix.Setrlimit` 手动解除 memlock 限制。

### v0.2.1 — 修复兼容模式在无 BTF 内核上的加载失败

**问题**：Oracle Linux 8 (UEK 5.4.x)、CentOS 7/8 等无 BTF 内核上，兼容模式报错 `apply CO-RE relocations: no BTF found`。  
**根因**：`bpf2go` 编译 `trace_legacy.bpf.c` 时，`vmlinux.h` 中的 `#pragma clang attribute push (__attribute__((preserve_access_index)))` 导致 `.BTF.ext` 生成 CO-RE 重定位记录。加载时尝试从内核加载 BTF 规范进行重定位，但目标内核没有 BTF。  
**修复**：为兼容版编译添加 `-DBPF_NO_PRESERVE_ACCESS_INDEX`，禁用 `preserve_access_index` 属性。`.BTF.ext` 中不再包含 `core_reloc` 段，但保留 `.BTF` 段用于 `cilium/ebpf` 解析 BTF-style map 定义。

### v0.2.0 — 双模式内核自适应

**核心特性**：运行时自动检测内核版本，≥ 5.8 使用现代模式（ringbuf + BTF/CO-RE），< 5.8 自动降级到兼容模式（perf buffer + legacy helper）。一套二进制同时覆盖 RHEL/CentOS/Oracle 7.6+ (3.10) ~ 最新 6.x 内核。
